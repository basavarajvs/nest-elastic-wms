### 🔍 Direct Answers to Your Questions

1. **Can hierarchy be maintained with `location_type`?**  
   **No.** `location_type` defines *what a location is used for* (e.g., `PALLET_RACK`, `CASE_PICK`, `FLOOR_STORAGE`, `STAGING`, `DOCK`). It is orthogonal to physical hierarchy. Two `CASE_PICK` locations can exist in completely different zones/aisles. Using it for hierarchy breaks routing, capacity planning, and inventory aggregation.

2. **How can hierarchy be maintained with a flat `location_code`?**  
   By using a **semantic/delimited naming convention** combined with optional structural columns:
   - Format: `ZONE-AISLE-BAY-LEVEL-LOC` (e.g., `A-01-B-02-L-03-01`)
   - Store as `location_code @unique`
   - Add `zone_id` (FK to `WarehouseZone`) for coarse grouping
   - Add `path` or `parent_location_id` if you need tree queries later
   - The string itself encodes the hierarchy; parsing it (`split('-')`) gives you zone/aisle/bay/level without extra FK tables.

3. **Other viable solutions?**
   - **Self-referencing tree**: `location.parent_id? @relation` (flexible depth, Prisma-friendly, but requires cycle prevention & careful indexing)
   - **PostgreSQL `ltree` extension**: `path ltree` (fast hierarchical queries, but adds extension dependency & raw SQL in Prisma)
   - **Full explicit hierarchy** (`Aisle → RackRow → Bay → Level → Location`): Strict referential integrity, matches `wms.sql` exactly, but heavy for initial scaffolding.

```sql
ALTER TABLE multitenant.resource_quotas ENABLE ROW LEVEL SECURITY;
ALTER TABLE multitenant.warehouse_facilities ENABLE ROW LEVEL SECURITY;
ALTER TABLE multitenant.warehouse_zones ENABLE ROW LEVEL SECURITY;

-- Create composite indexes for RLS performance
CREATE INDEX idx_rq_tenant ON multitenant.resource_quotas USING btree (tenant_id);
CREATE INDEX idx_wf_tenant ON multitenant.warehouse_facilities USING btree (tenant_id);
CREATE INDEX idx_wz_tenant ON multitenant.warehouse_zones USING btree (tenant_id);
CREATE INDEX idx_sl_tenant ON multitenant.storage_locations USING btree (tenant_id);

-- USING + WITH CHECK policies
DO $$
DECLARE
    tables text[] := ARRAY['resource_quotas', 'warehouse_facilities', 'warehouse_zones', 'storage_locations'];
    tbl text;
BEGIN
    FOREACH tbl IN ARRAY tables
    LOOP
        EXECUTE format('
            CREATE POLICY tenant_isolation_%1$s ON multitenant.%1$s
            USING (tenant_id = current_setting(''app.tenant_id'')::uuid);
            CREATE POLICY tenant_insert_%1$s ON multitenant.%1$s
            WITH CHECK (tenant_id = current_setting(''app.tenant_id'')::uuid);
        ', tbl);
    END LOOP;
END $$;
```

STEP E1 — Migration Script for RLS

```sql
-- Enable RLS on all WMS tables
ALTER TABLE multitenant.resource_quotas ENABLE ROW LEVEL SECURITY;
ALTER TABLE multitenant.warehouse_facilities ENABLE ROW LEVEL SECURITY;
ALTER TABLE multitenant.warehouse_zones ENABLE ROW LEVEL SECURITY;
ALTER TABLE multitenant.storage_locations ENABLE ROW LEVEL SECURITY;

-- Create composite indexes for RLS performance
CREATE INDEX idx_rq_tenant ON multitenant.resource_quotas USING btree (tenant_id);
CREATE INDEX idx_wf_tenant ON multitenant.warehouse_facilities USING btree (tenant_id);
CREATE INDEX idx_wz_tenant ON multitenant.warehouse_zones USING btree (tenant_id);
CREATE INDEX idx_sl_tenant ON multitenant.storage_locations USING btree (tenant_id);

-- USING + WITH CHECK policies
DO $$
DECLARE
    tables text[] := ARRAY['resource_quotas', 'warehouse_facilities', 'warehouse_zones', 'storage_locations'];
    tbl text;
BEGIN
    FOREACH tbl IN ARRAY tables
    LOOP
        EXECUTE format('
            CREATE POLICY tenant_isolation_%1$s ON multitenant.%1$s
            USING (tenant_id = current_setting(''app.tenant_id'')::uuid);
            CREATE POLICY tenant_insert_%1$s ON multitenant.%1$s
            WITH CHECK (tenant_id = current_setting(''app.tenant_id'')::uuid);
        ', tbl);
    END LOOP;
END $$;
```

### 🔍 Why Prompt 18? (Strategic Placement)
This is deliberately scheduled **after** the core operational modules (Inventory, Inbound/GRN, Outbound/Orders, Transfers) for three critical reasons:
1. **Entity Dependency**: The engines don't exist in a vacuum. They govern *real domain objects*. We need `products`, `inventory_on_hand`, `goods_receipts`, and `sales_orders` in the database before we can build rules that say *"If product is hazmat, route to Zone X"* or state machines that control *"ASN → GRN → Putaway"* lifecycles.
2. **Manhattan WMS Alignment**: In enterprise WMS, configuration comes after base functionality. Manhattan separates "base logic" from "client-specific configuration." We'll hardcode sane defaults in Prompts 14–17, then abstract them into configurable engines in Prompt 18.
3. **JSON/XML Storage Strategy**: The customization engine requires dedicated Prisma tables (`wms_state_machines`, `wms_rules`, `wms_processes`) to store tenant-specific definitions. Building this after core entities ensures the schema correctly references existing tables.

STEP B1 — Order Service & Status Derivation
==============================================
Create `src/outbound/order.service.ts`
- `create(dto, tenantId)`: Validates line items, checks quota, emits `order.created`. Status: `CREATED`.
- `validate(orderId, tenantId)`: Checks address, payment/terms, product active status. Transitions to `VALIDATED`.
- `syncStatusFromLines(orderId, tenantId)`: **Manhattan Rule** — Header status derived from line aggregation:
  `VALIDATED` → all lines `ALLOCATED`
  `WAVED` → any line `RELEASED`/`PICK_IN_PROGRESS`
  `PICKED` → all lines `PICKED`
  `PACKED` → all lines `PACKED`
  `SHIPPED` → all lines `SHIPPED`
  Never sets status directly; always computes from lines.

STEP B2 — Allocation Engine (Soft → Hard)
============================================
Create `src/outbound/allocation.service.ts`
- `softAllocate(orderId, tenantId)`:
  1. Fetch `validated` lines
  2. Query `InventoryOnHand` sorted by FEFO (`expiryDate ASC`), then `receivedDate ASC`
  3. Create `InventoryAllocation` records with `status: SOFT`, `expiresAt: now() + 24h`
  4. Update `InventoryOnHand.quantityAllocated`
- `hardAllocate(orderId, tenantId)`:
  1. Convert `SOFT` → `HARD_ALLOCATED` before wave release
  2. Lock specific `lotId` + `locationId`
  3. Emit `allocation.hard_locked`
- `releaseAllocation(allocationId, tenantId)`: Reverts `quantityAllocated`, marks `CANCELLED`.

STEP B3 — Wave Planning & Task Generation
============================================
Create `src/outbound/wave.service.ts`
- `create(criteria, tenantId)`: Filters orders by `carrier`, `deliveryDate`, `client`, `priority`.
- `releaseWave(waveId, tenantId)`:
  1. Validates `hardAllocate` status for all lines
  2. Generates `PickingTask` records ordered by `location.pickSequenceNumber` (S-route optimization)
  3. Sets wave status: `RELEASED`
  4. Updates order lines to `RELEASED`
  5. Emits `wave.released` → triggers RF push notifications

STEP B4 — RF Picking Execution Service
=========================================
Create `src/outbound/picking.service.ts`
- `confirmPick(taskId, actualQty, tenantId)`:
  1. Validates `task.status IN (ASSIGNED, IN_PROGRESS)`
  2. If `actualQty < task.quantityToPick` → `status: SHORT`, create backorder line
  3. Decrements `InventoryOnHand.quantityOnHand`, moves to `quantityPicked`
  4. Creates/updates LPN status: `PICKED`
  5. Updates `PickingTask.quantityPicked`, `status: COMPLETED`
  6. Emits `picking.completed` → updates line status → triggers header sync

STEP B5 — Packing & Shipping Services
========================================
Create `src/outbound/packing.service.ts`
- `startSession(userId, stationCode, tenantId)`: Creates session, sets `PACKING_ACTIVE`
- `scanIntoContainer(sessionId, containerCode, pickedLpnId, tenantId)`: Validates LPN is `PICKED`, attaches to container
- `closeSession(sessionId, tenantId)`: Validates all orders packed, generates `OUTBOUND_SHIPMENT` records, prints labels

Create `src/outbound/shipping.service.ts`
- `assignToLoad(shipmentId, loadId, tenantId)`: Validates carrier match, dock availability
- `confirmDispatch(loadId, tenantId)`: Sets shipment `SHIPPED`, updates `InventoryTransaction` type: `SHIP`, decrements `quantityOnHand`, clears `quantityPicked`

  2. Runs threshold routing:
     - `varianceValue <= autoThreshold` → Auto-approve, post adjustment
     - `autoThreshold < varianceValue <= supervisorThreshold` → `ApprovalLevel.SUPERVISOR`
     - `> supervisorThreshold` → `ApprovalLevel.MANAGER`
  3. Updates `status: RECONCILED`. Emits `count.completed`.

STEP B1 — Stock on Hand Service

Create `src/reports/stock-on-hand.service.ts`
Method: `extract(params: ReportParams, tenantId): Promise<AsyncGenerator<Record>>`
- Uses `$queryRaw` for performance:
  ```sql
  SELECT 
    p.product_code, p.product_name, p.velocity_class,
    loc.location_code, z.zone_code, loc.location_type,
    l.lot_number, l.expiry_date,
    u.uom_code,
    SUM(ioh.quantity_on_hand) as on_hand,
    SUM(ioh.quantity_allocated) as allocated,
    SUM(ioh.quantity_reserved) as reserved,
    (SUM(ioh.quantity_on_hand) - SUM(ioh.quantity_allocated) - SUM(ioh.quantity_reserved)) as available
  FROM inventory_on_hand ioh
  JOIN products p ON p.id = ioh.product_id AND p.tenant_id = $1
  JOIN storage_locations loc ON loc.id = ioh.location_id AND loc.tenant_id = $1
  JOIN warehouse_zones z ON z.id = loc.zone_id AND z.tenant_id = $1
  LEFT JOIN inventory_lots l ON l.id = ioh.lot_id
  JOIN units_of_measure u ON u.id = ioh.uom_id
  WHERE p.tenant_id = $1
    AND loc.facility_id = $2
    AND loc.is_active = true
  GROUP BY p.id, loc.id, l.id, u.id
  ORDER BY loc.location_code, p.product_code
  ```

### 🔌 Architectural Flow (Event-Driven & Non-Blocking)
```text
[RF Scanner] → POST /rf/inbound/receive/scan

      ↓
[EventEmitter2] → Async, fire-and-forget (does NOT block RF response)
      ↓
[WorkflowOrchestratorListener] → Catches event → Checks tenant config cache
      ↓

   ├─▶ [JDM Rules]   → Evaluates `{qcRequired: true, vendorRisk: 'HIGH'}`
   └─▶ [XState SM]   → Transitions `GRN` machine: `RECEIVING → QC_PENDING`
      ↓
All execution logged to `wms_execution_instances`. Failures → BullMQ retry/DLQ.
```

#### 1. Domain Event Emission (Inside `GRNService`)

#### 2. Workflow Orchestrator Listener (Built in Prompt 18)
```ts
// src/customization/listeners/workflow-orchestrator.listener.ts
@Injectable()
export class WorkflowOrchestratorListener {
  constructor(
    private readonly orchestrator: WorkflowOrchestratorService,
    private readonly redis: RedisService,
  ) {}

  @OnEvent('asn.received')
  async handleAsnReceived(payload: AsnReceivedEvent) {
    // 1. Idempotency check (prevent double-execution on retries)
    const execKey = `wms:exec:asn.received:${payload.tenantId}:${payload.asnId}`;
    const acquired = await this.redis.set(execKey, '1', 'EX', 3600, 'NX');
    if (!acquired) return; // Already processing

    // 2. Load tenant-specific workflow config (cached in Redis)
    const config = await this.orchestrator.getTenantWorkflowConfig(payload.tenantId, 'asn.received');
    if (!config) return; // Tenant has no workflow → zero overhead

    // 3. Execute engines async (BPMN + JDM + XState)
    await this.orchestrator.execute('asn_inbound_flow', 'ASN', payload.asnId, payload, config);
  }
}
```

```json
  }

```

### ✅ Why This Design Fits Manhattan WMS & RF Operations
| Concern | Solution |
|---------|----------|
| **RF Performance** | Events are `fire-and-forget`. RF gets `<50ms` response. Workflows run in background BullMQ threads. |
| **Tenant Isolation** | Each tenant's engine config is cached per-tenant. Cross-tenant bleed is impossible. RLS enforced on `wms_execution_instances`. |
| **Rule-Workflow-Process Synergy** | BPMN orchestrates the flow → JDM evaluates conditions at decision gates → XState manages entity lifecycle. All three run in the same execution context. |
| **Failure Safety** | If BPMN/JDM/XState throws, the transaction is retried 3x with exponential backoff. Critical RF flows are never blocked by engine failures. |
| **Audit Compliance** | Every engine step logs to `wms_execution_instances.contextJson`. Full trace: `who triggered → what evaluated → what transitioned → result`. |

STEP C1 — BullMQ Repeat-Job Deduplication
============================================
By default, `repeat` jobs spawn on *every* worker. In multi-pod deployments, this causes duplicate executions.
Fix via `jobId` + `repeat` consistency:
```ts
await this.queue.add('tenant-health-check', { tenantId }, {
  jobId: `cron:tenant-health:${tenantId}`, // Unique across cluster
  repeat: { pattern: '0 */30 * * * *', tz: 'UTC' },
  removeOnComplete: true
});
```
- BullMQ guarantees exactly-one execution per `jobId` across all workers
- Add `shared` queue config: `limiter: { groupKey: 'tenantId' }` for per-tenant rate limiting

STEP C2 — Redis Pub/Sub for Cross-Instance Invalidation
========================================================

  4. Migrations complete (`pg_advisory_lock` released)
  5. Cache warmed (optional)

  ```yaml

  readinessProbe: { httpGet: { path: '/health/ready', port: 3001 }, initialDelaySeconds: 30 }
  terminationGracePeriodSeconds: 30
  ```

  4. Exit 0

2. "RF scanners have high-throughput, low-latency requirements. Global rate limiting of 100 req/min will block legitimate RF workflows during peak receiving. Show how you configure `@Throttle('rf-scanner')` with IP + `X-Device-ID` composite keys in Redis, allows burst up to 500 req/sec for 10s, then throttles to 50 req/sec, preventing both abuse and legitimate workflow interruption."
3. "OpenAPI spec export (`/api/docs/openapi.json`) is used by partners to generate webhooks. If WMS schema changes (e.g., `Product` adds `trackSerial`), partners' old integrations break. Show how you implement OpenAPI versioning (`/api/v1/docs/openapi.json`, `/api/v2/docs/openapi.json`), deprecate old specs via `@Deprecated()` tags, and emit `api.spec.changed` webhook events 30 days before breaking schema changes go live."
4. "Fastify's `fastify-helmet` CSP blocks inline scripts, but Swagger UI requires inline JS/CSS. Show how you configure `contentSecurityPolicy.directives` to allow `script-src: 'self' 'unsafe-inline' 'nonce-<dynamic>'` specifically for `/api/docs` routes, while maintaining strict `default-src 'self'` for all other API endpoints, balancing Swagger functionality with production security."
5. "Multi-tenant SaaS means `X-Tenant-Code` header is required on every request. Swagger's default auth only supports JWT. Show how you extend `DocumentBuilder` to add a custom security scheme `TenantCodeAuth` that auto-populates `X-Tenant-Code` in Swagger UI, validates it against Redis cache before allowing 'Try It Out', and returns `401 Tenant Not Found` if invalid, preventing accidental cross-tenant testing in UI."
6. "Rate limiting uses Redis, but if Redis cluster goes down, Fastify defaults to blocking all requests (fail-closed). Show how you implement `fail-open` fallback: if Redis `PING` fails, rate limiter switches to in-memory LRU cache per-process with reduced limits (50% of normal), logs `rate_limit.degraded`, and allows critical RF endpoints to bypass throttling entirely to maintain warehouse operations during cache outage."
